# Phase -1: Pre-Migration — Enhanced Model-View Pattern

**Goal:** Enhance `TComponentModel` with `effect()` and `memo()` primitives so that ALL logic can live in the Model, making Views pure render functions. Then migrate page editors and app shell to use the new pattern.

**Status:** Planned

**Why before Phase 0:** The enhanced model primitives will be used by API interface implementations. If we build interfaces on the old pattern, we'd have to refactor them later. Better to have the foundation right first.

---

## Design

### Problem

Current `TComponentModel` handles `useState` and `useCallback` well — state lives in `TComponentState`, methods are class methods. But it has no answer for:

- **`useEffect`** — side effects that run when specific values change (e.g., start file watcher when `filePath` prop changes). Currently these must stay in the View as React hooks.
- **`useMemo`** — cached computations that invalidate when dependencies change. Currently stays in the View or becomes uncached getters in the Model.

This means Views still contain logic — breaking the clean Model/View separation.

### Solution: Two New Primitives

Both built on a shared dependency-tracking foundation:

```typescript
// Shallow comparison — same algorithm React uses
function depsChanged(prev: any[] | undefined, next: any[]): boolean {
  if (!prev || prev.length !== next.length) return true;
  return prev.some((v, i) => !Object.is(v, next[i]));
}
```

#### `this.effect(callback, depsFactory)` — useEffect for Models

```typescript
class MyModel extends TComponentModel<State, Props> {
  init() {
    // Runs when this.props.filePath changes
    // Cleanup runs automatically before re-run and on unmount
    this.effect(
      () => {
        const watcher = new FileWatcher(this.props.filePath, this.onChange);
        return () => watcher.dispose();  // cleanup function
      },
      () => [this.props.filePath]  // deps factory — evaluated each cycle
    );

    // Effect with no deps — runs once (like useEffect(fn, []))
    this.effect(() => {
      window.addEventListener("resize", this.onResize);
      return () => window.removeEventListener("resize", this.onResize);
    });
  }
}
```

**How it works:**
- Effects are registered in `init()` (or `setProps()`)
- `setPropsInternal()` evaluates all effect deps on each render cycle
- If deps changed: run cleanup of previous execution, then run callback
- `onUnmountInternal()` runs all remaining cleanups
- No deps = runs once on init, cleanup on unmount (same as `useEffect(fn, [])`)

#### `this.memo(computeFn, depsFactory)` — useMemo for Models

```typescript
class MyModel extends TComponentModel<State, Props> {
  // Cached computation — recalculates only when deps change
  filteredItems = this.memo(
    () => this.props.items.filter(i => i.active),
    () => [this.props.items]
  );

  // In View: model.filteredItems.value
}
```

**How it works:**
- Returns an object with `.value` getter
- On `.value` access, checks if deps changed since last computation
- If changed: recompute, cache, return new value
- If same: return cached value
- Alternatively: evaluated eagerly in `setPropsInternal()` alongside effects

### Auto Init/Dispose in `useComponentModel`

Currently the View must manually wire lifecycle:

```typescript
// Current — boilerplate in every View
useEffect(() => {
  model.init();
  return () => model.dispose();
}, []);
```

Enhanced `useComponentModel` does this automatically:

```typescript
// New — no lifecycle boilerplate needed
export function useComponentModel<T, P, M extends TComponentModel<T, P>>(
  props: P, model: ..., defaultState?: T
): M {
  const controlModel = useModel(model, TComponentState, defaultState);
  controlModel.setPropsInternal(props);

  if (controlModel.isFirstUse) {
    controlModel.init?.();           // auto-call init on first use
    controlModel.isFirstUse = false;
  }

  useEffect(() => () => {
    controlModel.dispose?.();        // auto-call dispose on unmount
    controlModel.onUnmountInternal();
  }, []);

  return controlModel;
}
```

Models that need init/dispose just define the methods. No useEffect boilerplate in Views.

### What the View Becomes

```typescript
// Before — logic in View
function ImageView(props: Props) {
  const model = useComponentModel(props, ImageViewModel, defaultState);
  const { zoom, src } = model.state.use(s => ({ zoom: s.zoom, src: s.src }));

  useEffect(() => { model.init(); return () => model.dispose(); }, []);
  useEffect(() => { model.updateFitScale(); }, [src]);

  const displaySize = useMemo(() => calcSize(zoom, src), [zoom, src]);

  return <img style={{ width: displaySize.w }} src={src} />;
}

// After — View is pure rendering
function ImageView(props: Props) {
  const model = useComponentModel(props, ImageViewModel, defaultState);
  const { zoom, src } = model.state.use(s => ({ zoom: s.zoom, src: s.src }));

  return <img style={{ width: model.displaySize.value.w }} src={src} />;
}
```

### What to Migrate

**Page editors and app shell** — these are the components that will become API interface objects or interact heavily with them:

| Area | Models | Views | Priority |
|------|--------|-------|----------|
| Core framework | TComponentModel, useComponentModel | — | Must be first |
| App shell | MainPageModel, EventHandlerModel | MainPage, EventHandler | High |
| Tabs | TabsModel, PageTabModel | PageTabs, PageTab | High |
| Text editor | TextEditorModel, ScriptPanelModel | TextEditor, ScriptPanel | High |
| Grid editor | GridPageModel | GridEditor | Medium |
| Notebook editor | NotebookEditorModel, NoteItemViewModel | NotebookEditor, NoteItemView | Medium |
| Todo editor | TodoEditorModel | TodoEditor | Medium |
| Link editor | LinkEditorModel | LinkEditor | Medium |
| Image viewer | ImageViewModel | BaseImageView | Medium |
| Markdown view | MarkdownViewModel | MarkdownView | Medium |
| Compare editor | CompareEditorModel | CompareEditor | Low |
| Menu bar | MenuBarModel | MenuBar | Low |

### What NOT to Migrate (Skip)

Reusable UI components — they work fine and aren't part of the API layer:

- AVGridModel, RenderGridModel, RenderFlexGridModel (data grid internals)
- TreeViewModel, FileExplorerModel (tree components)
- MinimapModel, PopupMenuModel, PathInputModel (utility components)

These can be migrated later if needed, but they're not on the critical path.

## Tasks

### -1.0. Enhance TComponentModel

**Files:** `/src/renderer/core/state/model.ts`

Add to `TComponentModel`:
- `effect(callback, depsFactory?)` — register a deps-tracked side effect
- `memo(computeFn, depsFactory)` — register a deps-tracked cached computation
- Effect evaluation in `setPropsInternal()`
- Effect cleanup in `onUnmountInternal()`
- Auto `init()` / `dispose()` calls in `useComponentModel` hook

Add shared utility:
- `depsChanged(prev, next)` — shallow dependency comparison

Update documentation:
- `/doc/standards/model-view-pattern.md` — document new primitives
- `/doc/architecture/state-management.md` — update TComponentModel section

**Acceptance criteria:**
- [ ] `this.effect()` works with deps (re-runs on change) and without deps (runs once)
- [ ] `this.memo()` returns cached value, recomputes only when deps change
- [ ] Effect cleanup runs on deps change and on unmount
- [ ] `useComponentModel` auto-calls `init()` and `dispose()`
- [ ] Existing components still work without changes (backward compatible)
- [ ] No React hooks needed in Views for lifecycle or derived state

### -1.1. App Shell

**Models:** MainPageModel, EventHandlerModel
**Views:** MainPage.tsx, EventHandler.tsx

Current useEffect patterns to move into models:
- MainPageModel: `init()` subscribes to window maximize/zoom events, `destroy()` cleans up
- EventHandlerModel: `init()` attaches document event listeners (contextmenu, dragover, drop), `dispose()` removes them

**What changes:**
- Move `useEffect(() => { model.init(); return () => model.destroy(); }, [])` into auto init/dispose
- Any additional useEffects become `this.effect()` calls in `init()`

---

### -1.2. Tabs & Navigation

**Models:** TabsModel, PageTabModel, MenuBarModel
**Views:** PageTabs.tsx, PageTab.tsx, MenuBar.tsx

Current patterns:
- TabsModel: `useEffect(() => model.destroy, [])` — cleanup only
- MenuBarModel: `useEffect(() => { model.init() }, [deps])` — init with dependencies

**What changes:**
- Auto init/dispose replaces useEffect boilerplate
- MenuBar's dep-based init becomes `this.effect()` in model

---

### -1.3. Text Editor

**Models:** TextEditorModel, ScriptPanelModel
**Views:** TextEditor.tsx, ScriptPanel.tsx

Current patterns:
- TextEditor: `useEffect` subscribes to pagesModel.onFocus, returns cleanup that calls `editorModel.onDispose()`
- ScriptPanel: model has `dispose()` that cleans up Monaco selection listener

**What changes:**
- Focus subscription becomes `this.effect()` in TextEditorModel
- Monaco listener cleanup tracked in `dispose()`

### -1.4. Grid Editor

**Models:** GridPageModel
**Views:** GridEditor.tsx

Current patterns:
- GridPageModel has `init()` and `dispose()` (state subscription)
- View has minimal useEffect

**What changes:**
- Auto init/dispose
- State subscription in `init()` can use `this.effect()` or stay as direct subscription tracked in DisposableCollection

### -1.5. Notebook & Note Items

**Models:** NotebookEditorModel, NoteItemViewModel
**Views:** NotebookEditor.tsx, NoteItemView.tsx

Current patterns:
- NotebookEditorModel: `init()` subscribes to state changes, `dispose()` unsubscribes
- NoteItemViewModel: `init()` + `dispose()` — View has `useEffect(() => { model.init(); return () => model.dispose(); }, [])`

**What changes:**
- Auto init/dispose removes View boilerplate
- State subscriptions become tracked disposables

### -1.6. Todo & Link Editors

**Models:** TodoEditorModel, LinkEditorModel
**Views:** TodoEditor.tsx, LinkEditor.tsx

Current patterns:
- Both: `init()` subscribes to state for debounced serialization, `dispose()` unsubscribes
- Very similar pattern — can be migrated together

**What changes:**
- Auto init/dispose
- State subscriptions become tracked disposables

---

### -1.7. Image Viewer

**Models:** ImageViewModel
**Views:** BaseImageView.tsx

Current patterns — **most complex**, 3 useEffects:
1. `useEffect(() => { model.init(); return () => model.dispose(); }, [])` — lifecycle
2. `useEffect(() => { model.updateFitScale(); }, [src])` — recalculate on image change
3. Additional effect for src changes

**What changes:**
- Lifecycle → auto init/dispose
- Fit scale recalculation → `this.effect(() => this.updateFitScale(), () => [this.state.get().src])`
- This is the best demonstration of `this.effect()` value

---

### -1.8. Markdown View

**Models:** MarkdownViewModel
**Views:** MarkdownView.tsx

Current patterns:
- `useMemo` for syntax highlighting
- `useEffect` for search match navigation

**What changes:**
- Syntax highlighting → `this.memo()` in model
- Search navigation → `this.effect()` in model
