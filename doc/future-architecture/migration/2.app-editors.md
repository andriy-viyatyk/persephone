# 2. `app.editors` — IEditorRegistry

**Phase:** 1b (Independent — no dependencies on other interface objects)

**Status:** Complete

**API Reference:** [../api-reference/editors.md](../api-reference/editors.md) (to be written)

## Summary

The editor registry is a **complex object** with heavy internal logic (registration, lazy loading, file resolution algorithms). Per the migration strategy's "Thin Wrappers for Complex Objects" design decision, the facade in `/api/editors.ts` is the **correct pattern** — it exposes a read-only subset (`getAll`, `getById`, `resolve`, `resolveId`, `getSwitchOptions`) while internal logic stays encapsulated in `/editors/registry.ts`.

This is different from the "thin wrapper" anti-pattern because the wrapper delegates to code in the **same module family** (editors), not to a distant old store. No code movement is needed.

---

## What Was Done

- `IEditorRegistry` and `IEditorInfo` interfaces defined in `/api/types/editors.d.ts`
- Facade implementation in `/api/editors.ts` delegates to `editorRegistry`
- `app.editors` registered on the `app` singleton
- Internal details (`register()`, `loadModule()`, `acceptFile()`) are NOT exposed
- API reference document created
- **No consumer migration needed** — consumers that use `editorRegistry` directly need internal details (registration, module loading) that the public interface intentionally doesn't expose

---

## Key Decision: Why No Code Movement

The editor registry has 10 consumer files, but they use internal methods (`register()`, `loadModule()`, `validateForLanguage()`, `getPreviewEditor()`) that are intentionally NOT part of the public interface. Moving logic into `/api/editors.ts` would either:
1. Expose internals that shouldn't be public, or
2. Create a file that's both the public API and the internal implementation — defeating the purpose of separation

The facade pattern is correct here. The interface IS the public API surface; the implementation stays internal to the editors module.
