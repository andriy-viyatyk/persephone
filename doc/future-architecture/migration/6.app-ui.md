## What

Dialogs and notifications:
- Confirmation dialog (`confirm`)
- Text input dialog (`input`)
- Password dialog (`password`)
- Toast notifications (`notify`)

This phase also moves `features/dialogs/` → `ui/dialogs/` — the first creation of the `/ui/` presentation layer from the migration plan's target structure.

---

## Implementation

### Approach: Wrapping, Not Absorption

Dialog components are complex React UI (models + views + styling). Per the migration plan's design decision on "Thin Wrappers for Complex Objects": the interface is a thin public API over a complex internal implementation. Components stay in `ui/dialogs/`.

`api/ui.ts` wraps the existing `show*` functions with a cleaner interface. Each method uses **dynamic `import()`** internally to preserve code splitting — dialog code only loads when the dialog is actually shown.

### Architecture

```
api/ui.ts — UserInterface implements IUserInterface
├── confirm(message, options?) → dynamic import("../ui/dialogs/ConfirmationDialog")
│                                  → showConfirmationDialog({ message, ...options })
│                                  → maps undefined to null
├── input(message, options?) → dynamic import("../ui/dialogs/InputDialog")
│                               → showInputDialog({ message, ...options })
│                               → maps undefined to null
├── password(options?) → dynamic import("../ui/dialogs/PasswordDialog")
│                         → showPasswordDialog(options)
│                         → maps undefined to null
└── notify(message, type?) → alertsBarModel.addAlert(message, type ?? "info")
                              → returns Promise (resolves with "clicked" or undefined)
```

### EventHandler special case: `notify()` with click detection

EventHandler uses `await ui.notify(...)` to detect when the user clicks the update-available notification:

```typescript
const closeResult = await ui.notify(
    `New version ${version} is available! Click to open About page.`,
    "info",
);
if (closeResult === "clicked") {
    showAboutPage();
}
```

This works because `notify()` returns a `Promise<string | undefined>` — it resolves with `"clicked"` when the user clicks the notification body, or `undefined` when dismissed/auto-closed.

## Design Decisions

1. **`null` instead of `undefined`** — Cancel/dismiss returns `null` (not `undefined`). Consistent with `app.fs` convention from Phase 2.

2. **`notify()` returns `Promise<string | undefined>`** — Changed from original plan's `void`. Allows callers to detect clicks (`"clicked"`) or dismissal (`undefined`). Most callers ignore the promise (fire-and-forget); EventHandler awaits it for click-to-navigate behavior.

3. **Dynamic imports in methods** — Each dialog method lazily imports the show function, preserving code splitting. Dialog code only loads when the dialog is actually used.

4. **`alertsBarModel` exported, not moved** — The alert model stays in `ui/dialogs/alerts/AlertsBar.tsx` (near its rendering component). `api/ui.ts` imports it directly.

5. **Folder move included** — Since all consumer imports were being updated for the API migration anyway, moving `features/dialogs/` → `ui/dialogs/` was zero extra effort. First step of the `/ui/` layer from the target structure.
