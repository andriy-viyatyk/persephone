# Phase 4: `app.pages` — IPageCollection

**Goal:** Move page management logic from `store/pages-store.ts` into the new `api/pages/` module using category-based submodel decomposition. Absorb `page-actions.ts` convenience functions. Wire into `app.pages` and the bootstrap sequence.

**Status:** Complete

**Architecture reference:** [/doc/architecture/pages-architecture.md](../../architecture/pages-architecture.md) — diagrams for bootstrap lifecycle, page lifecycle, submodel architecture.

---

## What Was Done

### US-049: Event Services + Downloads API (Phase 4a)

Implemented the 3-layer bootstrap sequence and extracted event handling from React components into internal services.

- ✅ Removed `MainPage useEffect` — Bootstrap now in `renderer.tsx` before React renders
- ✅ Removed `EventHandler` component — Event subscriptions moved to 4 internal services
- ✅ Added `app.initServices()` — Layer 1: Load 8 APIs in parallel (including downloads)
- ✅ Added `app.initPages()` — Layer 2: Restore pages + handle CLI args
- ✅ Added `app.initEvents()` — Layer 3: Subscribe to all events (GlobalEventService, KeyboardService, WindowStateService, RendererEventsService)
- ✅ Added `api.windowReady()` — Signal main process that window is fully initialized
- ✅ Explicit 3-phase sequence orchestrated from `app.ts`, not React lifecycle
- ✅ Guard code on all `initXxx()` methods prevents re-initialization

### US-050: Pages API (Phase 4b)

Moved page management from `store/pages-store.ts` into 5 category submodels under `api/pages/`.

- ✅ Created `PagesModel` (base state), `PagesLifecycleModel`, `PagesNavigationModel`, `PagesLayoutModel`, `PagesPersistenceModel`, `PagesQueryModel`
- ✅ All page logic moved from `pages-store.ts` — not thin wrappers
- ✅ `page-actions.ts` convenience functions absorbed into `PagesModel` methods (`openUrlInBrowserTab`, `openImageInNewTab`, `showBrowserPage`, `showSettingsPage`, `showAboutPage`)
- ✅ `IPageCollection` added to `IApp` interface in `types/app.d.ts`
- ✅ All 29 consumer files migrated from `store/pages-store` and `store/page-actions` to `api/pages`
- ✅ `store/pages-store.ts` deleted (was re-export bridge)
- ✅ `store/page-actions.ts` deleted (was re-export bridge)
- ✅ `store/index.ts` cleaned — only exports `menuFolders` and language utilities
- ✅ Removed 100ms timeout hack from `fs.ts` — bootstrap now properly awaits `fs.wait()`
- ✅ Fixed `api.windowReady()` missing call in renderer.tsx (critical for multi-window page transfer)

## Design Decisions (Final)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Submodel pattern | Category-based (lifecycle, navigation, layout, persistence, query) | Matches AVGridModel pattern, keeps files focused |
| Public interface | `IPageCollection` + `IPageInfo` in `.d.ts` | Script-facing, minimal — no internal methods |
| Multi-window ops | `movePageIn`/`movePageOut` stay internal | IPC-driven, not for scripts |
| Error handling | Throw on user actions, catch on initialization | Initialization notifies user and creates fallback |
| Persistence | Automatic debounced save, explicit `save()` for quit | Reliable without user intervention |
| Window ready signal | `api.windowReady()` after `initEvents()` | Main process waits before sending IPC events to window |
| `IApp` typing | `App` class does NOT implement `IApp` | `IApp` is script-facing (.d.ts); `App` has internal methods (`init`, `initServices`, etc.) with richer types |
