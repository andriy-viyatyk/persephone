# 5. `app.window` — IWindow

**Phase:** 2b (Window — depends on IPC event system)

**Status:** Complete

**API Reference:** [../api-reference/window.md](../api-reference/window.md)

---

## What

Window management API wrapping IPC window endpoints:
- Window actions: minimize, maximize, restore, close
- Zoom: zoom in/out, reset, current level
- State: isMaximized (reactive)
- Multi-window: openNew

Phase 2 subset only. Methods that require `app.pages` (like `showWindowPage`, page-moving between windows) are deferred to Phase 4.

### Delegation

- `minimize()` / `maximize()` / `restore()` / `close()` → `api.minimizeWindow()` / `api.maximizeWindow()` / `api.restoreWindow()` / `api.closeWindow()`
- `zoom(delta)` → `api.zoom(delta)`
- `resetZoom()` → `api.resetZoom()`
- `openNew(filePath?)` → `api.openNewWindow(filePath)`
- `isMaximized` → cached from `rendererEvents.eWindowMaximized` subscription
- `zoomLevel` → cached from `rendererEvents.eZoomChanged` subscription

### Reactive State

The `Window` class subscribes to IPC events in its constructor:
- `eWindowMaximized` → updates `_isMaximized`
- `eZoomChanged` → updates `_zoomLevel`

Getters return cached values synchronously — same pattern as `MainPageModel` but at the API level.

---

## Design Decisions

1. **Phase 2 subset** — only window actions, zoom, and openNew. Page-moving between windows deferred to Phase 4 (requires `app.pages`).
2. **Reactive properties** — `isMaximized` and `zoomLevel` cached from event subscriptions, returned synchronously.
3. **Fire-and-forget actions** — window actions (`minimize`, `maximize`, etc.) don't await IPC responses.
4. **No event cleanup** — subscriptions live for the lifetime of the window. Since each window has its own `app` instance, cleanup happens when the window closes.

---

## Deferred to Phase 4

The following will be added when `app.pages` is implemented:
- `showWindowPage(windowIndex, pageId)` — navigate to a specific page in another window
- Page-moving between windows (requires page identity via `app.pages`)
