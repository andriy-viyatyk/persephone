# Phase 0: Infrastructure

**Goal:** Create the folder structure, base types, async initialization, and wiring so that `app` exists as a global in scripts (fully initialized before React renders).

**Status:** Planned

---

## Design Decisions

### Circular Reference Avoidance

Wiring everything into a single `app` object could create circular imports. We avoid this by design:

**Rule: Each wrapper imports its backing store directly, never through `app.ts`.**

`app.ts` is a **leaf assembler** — it imports all wrappers and composes them. Nothing imports `app.ts` except the bootstrap entry point and ScriptContext. The dependency arrows all point one way:

```
bootstrap (index.tsx) → app.ts → settings.ts → app-settings.ts (store)
                                → editors.ts  → registry.ts
                                → recent.ts   → recent-files.ts (store)
                                → fs.ts       → files-store.ts (store)

ScriptContext.ts → app.ts (only other consumer)
```

**What about cross-interface calls?** If one interface implementation needs another (e.g., `recent.ts` needs `filesModel` for persistence), it imports the backing store directly — not through `app.ts`:

```typescript
// recent.ts — OK, no circle
import { recentFiles } from "../store/recent-files";  // imports store, not app

// recent.ts — WRONG, would create circle
import { app } from "./app";  // never do this from a wrapper
```

**Where could problems arise?**
- `app.ui` (Phase 3) — might need to show React dialog components. Solved by having `ui.ts` import a dialog **service** (imperative API), not React components directly.
- Editor-specific interfaces (Phase 5) — live near their editors but get registered on `IPage`. Solved by lazy `asText()` / `asBrowser()` accessors that use `import()`.

**Summary:** With the wrapper-imports-store pattern, circular references are structurally impossible for Phases 0–2 and manageable for later phases.

### Async Initialization

The `app` object cannot be created synchronously — several services depend on async operations:

| Service | Async dependency |
|---------|-----------------|
| `app.fs` | Resolve data paths via IPC (`getCommonFolder`) |
| `app.settings` | Read `appSettings.json` from disk |
| `app.recent` | Read `recentFiles.txt` from disk |
| Editor registry | Synchronous — no async needed |

**Current problem:** Stores fire-and-forget their async init at module load time. Components may render before settings are loaded. No guarantee of readiness.

**Solution: Explicit `app.init()` that completes before React renders.**

```typescript
// /src/renderer/api/app.ts

class App implements IApp {
  private _initialized = false;

  async init(): Promise<void> {
    // All independent init in parallel for speed
    await Promise.all([
      this._settings.init(),    // read appSettings.json
      this._fs.init(),          // resolve data paths via IPC
      this._recent.init(),      // read recentFiles.txt
    ]);

    // Dependent init (if any) runs after
    // ...

    this._initialized = true;
  }
}
```

```typescript
// /src/renderer/index.tsx — bootstrap

import { app } from "./api/app";

async function bootstrap() {
  await app.init();        // app is fully ready

  // NOW start React — safe to call app.settings.theme etc.
  const root = createRoot(document.getElementById("root"));
  root.render(<AppContent />);
}

bootstrap();
```

**Benefits:**
- **Parallel init** — all independent async work runs concurrently via `Promise.all`
- **Guaranteed ready** — React rendering starts only after `app` is fully initialized
- **No race conditions** — components can safely read `app.settings.theme` during first render
- **Measurable** — can log init duration for performance monitoring

**Notes:**
- The `init()` method is internal (not exposed to scripts) — scripts run long after initialization
- Current stores (`filesModel.init()`, `appSettings` constructor) already have async init — we formalize and parallelize it
- If a future interface needs no async init, it simply doesn't participate in `Promise.all`

### Type Declarations — Single Source for TypeScript + Monaco

Interface declarations (`.d.ts` files) serve **three purposes** from a single source:

1. **TypeScript compilation** — implementation files import and implement these interfaces. The compiler enforces correctness.
2. **Monaco autocompletion** — copied to `assets/editor-types/` during build. Loaded via `addExtraLib()` for script IntelliSense.
3. **Documentation** — JSDoc comments on every interface/property/method become hover tooltips in the script editor.

**Current approach:** Hand-written `.d.ts` files live in `assets/editor-types/`. They are not type-checked — can drift from the actual implementation.

**New approach:** `.d.ts` files live in `/src/renderer/api/types/`. Implementations import and implement them. A build step copies them to `assets/editor-types/`.

```
                     /src/renderer/api/types/*.d.ts
                           (source of truth)
                          /                   \
                         /                     \
            TypeScript compiler            Build step: copy
            enforces implementations       to assets/editor-types/
                    |                              |
            /src/renderer/api/*.ts         /assets/editor-types/*.d.ts
            (implementations)              (loaded into Monaco at runtime)
```

**What this looks like:**

```typescript
// /src/renderer/api/types/settings.d.ts — the declaration

/** Application configuration. Read and write settings with change notifications. */
export interface ISettings {
  /** Current theme name */
  readonly theme: string;

  /** Get a setting value by key */
  get<T = any>(key: string): T;

  /** Set a setting value. Persisted automatically. */
  set<T = any>(key: string, value: T): void;

  /** Fires when any setting changes */
  readonly onChanged: IEvent<{ key: string; value: any }>;
}
```

```typescript
// /src/renderer/api/settings.ts — the implementation

import type { ISettings } from "./types/settings";

class Settings implements ISettings {
  // TypeScript enforces all properties/methods match the declaration
}
```

```typescript
// /src/renderer/api/types/index.d.ts — globals for scripts

import type { IApp } from "./app";
import type { IPage } from "./page";

declare global {
    /** The application object. Access all app functionality through this. */
    const app: IApp;

    /** The active page. Available in scripts that run in context of a page. */
    const page: IPage | undefined;
}

export {};
```

**Build integration:**

A Vite plugin or npm script:
1. Copies `src/renderer/api/types/*.d.ts` → `assets/editor-types/`
2. Generates `_imports.txt` from the file list (replacing the current hand-maintained list)

The existing `configure-monaco.ts` already loads from `assets/editor-types/` via `app-asset://` protocol — no changes needed there.

**Benefits:**
- No duplication — one set of `.d.ts` files
- Always in sync — TypeScript compiler catches drift
- JSDoc comments flow through to Monaco tooltips automatically
- Adding a new interface = add one `.d.ts` file, it appears in autocompletion

**Notes:**
- The `/types/` folder contains ONLY pure `.d.ts` declarations — no implementation code, no imports of implementation modules
- Helper types used only by implementation (like `wrapSubscription`) live in a separate utility file, not in `/types/`
- The current `assets/editor-types/page.d.ts` is replaced by `src/renderer/api/types/page.d.ts`

### Disposable Pattern — Unified Resource Cleanup

Any object that acquires resources (subscriptions, file watchers, event listeners, timers) must have a way to release them. Without a formal pattern, cleanup is ad-hoc and leaks happen silently.

**Current state — 4 inconsistent cleanup patterns in the codebase:**

| Pattern | Used by | Example |
|---------|---------|---------|
| `.dispose()` method | FileWatcher, TextPageModel, ScriptPanelModel, NavPanelModel | `this.fileWatcher.dispose()` |
| `.unsubscribe()` on object | Subscription class returns `{ unsubscribe() }` | `subscription.unsubscribe()` |
| Call returned function | Grid, Notebook, Todo, Link editors | `this.stateChangeSubscription?.()` |
| Monaco `IDisposable` | Monaco editor listeners | `this.selectionListenerDisposable?.dispose()` |

All four do the same thing — "clean up after yourself." We unify them under one interface.

**Solution: `IDisposable` interface + `DisposableCollection` utility.**

```typescript
// types/common.d.ts — the universal cleanup contract

/** An object that holds resources and must be cleaned up when no longer needed. */
interface IDisposable {
  dispose(): void;
}
```

One method, one contract. Matches Monaco's own `IDisposable`, so it's already familiar to the codebase.

**`IEvent.subscribe()` returns `IDisposable`** (not a raw function, not `{ unsubscribe() }`):

```typescript
interface IEvent<T> {
  subscribe(handler: (data: T) => void): IDisposable;
}
```

This means every subscription is a disposable — uniform cleanup regardless of the event source.

**`DisposableCollection` — group multiple disposables, dispose all at once:**

```typescript
// internal.ts — implementation utility (not exposed to scripts)

class DisposableCollection implements IDisposable {
  private items: IDisposable[] = [];

  /** Track a disposable for later cleanup */
  add<T extends IDisposable>(disposable: T): T {
    this.items.push(disposable);
    return disposable;  // allows: const sub = disposables.add(event.subscribe(...))
  }

  /** Dispose all tracked items and clear the list */
  dispose(): void {
    for (const item of this.items) {
      item.dispose();
    }
    this.items = [];
  }
}
```

**How implementations use it:**

```typescript
class Settings implements ISettings {
  private _disposables = new DisposableCollection();

  init() {
    // Every subscription is tracked automatically
    this._disposables.add(
      appSettings.onChanged.subscribe(() => { ... })
    );
    this._disposables.add(
      new FileWatcher(path, callback)
    );
  }

  dispose() {
    this._disposables.dispose(); // cleans up everything — one call
  }
}
```

No more forgetting to unsubscribe. No more tracking 5 separate fields.

**Script cleanup — automatic disposal:**

When a script subscribes to events, the ScriptContext collects all disposables created during execution and auto-disposes them when the script finishes:

```javascript
// User script — subscription is auto-cleaned when script ends
app.pages.onPageOpened.subscribe(page => {
  console.log("opened:", page.title);
});
```

The ScriptContext wraps event subscription to intercept `subscribe()` calls and track returned disposables. When the script finishes (or is stopped), all collected disposables are disposed automatically. No manual cleanup needed for simple scripts.

For long-running or persistent scripts (future), explicit disposal remains available.

**How it fits into object lifecycles:**

Every interface object that holds resources implements `IDisposable`:

```
IPage lifecycle:  Creating → Active → Closing → dispose() → all resources cleaned
ISettings:        init() → Active → dispose() → file watcher + subscriptions cleaned
```

The `app` object itself is NOT disposable — it lives for the window lifetime and is cleaned up by the process exit. But `IPage`, editor-specific interfaces, and any subscription-holding objects implement `IDisposable`.

**What this changes in Phase 0:**
- `IDisposable` added to `types/common.d.ts`
- `DisposableCollection` added to `internal.ts`
- `IEvent.subscribe()` returns `IDisposable` (not `() => void`)
- `wrapSubscription` adapter returns `IDisposable` from existing `Subscription`

**Benefits:**
- **Uniform cleanup** — one pattern everywhere (`.dispose()`)
- **Composable** — `DisposableCollection` groups multiple resources
- **Script-safe** — automatic cleanup prevents leaks from user scripts
- **Monaco-compatible** — matches Monaco's own `IDisposable`
- **Lifecycle-integrated** — fits naturally into page and object lifecycles

### 0.3. Define internal helpers

**`/src/renderer/api/internal.ts`** — implementation utilities (NOT copied to assets):

```typescript
import { Subscription } from "../core/state/subscription";
import type { IDisposable, IEvent } from "./types/common";

/**
 * Collects multiple disposables and disposes them all at once.
 * Use in interface implementations to track subscriptions and resources.
 */
export class DisposableCollection implements IDisposable {
  private items: IDisposable[] = [];

  /** Track a disposable. Returns it for inline use. */
  add<T extends IDisposable>(disposable: T): T {
    this.items.push(disposable);
    return disposable;
  }

  /** Dispose all tracked items and clear the list. */
  dispose(): void {
    for (const item of this.items) {
      item.dispose();
    }
    this.items = [];
  }
}

/**
 * Adapts an existing Subscription<T> to the IEvent interface.
 * The returned IEvent.subscribe() produces IDisposable objects.
 */
export function wrapSubscription<T>(subscription: Subscription<T>): IEvent<T> {
  return {
    subscribe(handler: (data: T) => void): IDisposable {
      const sub = subscription.subscribe(handler);
      return { dispose: () => sub.unsubscribe() };
    },
  };
}
```

**Notes:**
- `IEvent<T>` wraps existing `Subscription<T>` — no new event system needed
- `subscribe()` returns `IDisposable` — uniform cleanup via `.dispose()`
- `wrapSubscription` adapts the existing `{ unsubscribe() }` to `IDisposable`
- `DisposableCollection` groups multiple disposables for bulk cleanup
- `IApp` interface is defined in `types/app.d.ts` — not duplicated here
