# US-377: Ref Resolution Improvements

## Goal

Centralize and improve ref-based element resolution in the automation layer. Currently, `DOM.resolveNode` + `Runtime.callFunctionOn` logic is duplicated across 3 files. Extract a reusable `resolveRef()` utility and evaluate whether the current CDP approach is reliable enough, or needs improvement.

## Background

### Current implementation

Refs come from the accessibility snapshot (`snapshot.ts`), which uses `backendDOMNodeId` from CDP `Accessibility.getFullAXTree`. The snapshot formats them as `[ref=e123]` where `123` is the `backendDOMNodeId`.

Resolution logic is **duplicated in 3 places**:

**1. `commands.ts:66-76` — `callOnRef()`**
```typescript

    const backendNodeId = parseInt(ref.replace(/^e/, ""), 10);
    if (isNaN(backendNodeId)) throw new Error("Invalid ref: " + ref);
    const { object } = await cdp.send("DOM.resolveNode", { backendNodeId });
    if (!object?.objectId) throw new Error("Could not resolve ref: " + ref);
    await cdp.send("Runtime.callFunctionOn", {
        objectId: object.objectId,
        functionDeclaration: fn,
        awaitPromise: true,
    });
}
```
Used by: `browserClick` (ref path), `browserSelectOption` (ref path)

**2. `input.ts:192-211` — `focusElementByRef()`**
```typescript

    const backendNodeId = parseInt(ref.replace(/^e/, ""), 10);
    // ... same DOM.resolveNode + Runtime.callFunctionOn pattern
}
```
Used by: `typeText()` (ref path)

**3. `input.ts:240-252` — inside `fillInput()` ref branch**
```typescript

// ... same pattern again
```
Used by: `fillInput()` when called with ref instead of selector

### Problems with duplication

1. **Maintenance burden**: Same 5-line pattern in 3 places. Error handling inconsistencies possible.
2. **No central error messages**: Each copy has slightly different error text.
3. **No stale ref detection**: If the DOM node was removed since the snapshot, `DOM.resolveNode` throws. No helpful error message guiding the agent to re-snapshot.

### Playwright's approach (from source investigation)

Playwright uses a completely different architecture:
- Snapshot is generated by **injected JavaScript** (not CDP `Accessibility.getFullAXTree`)
- During snapshot, stores a `Map<ref, Element>` in-page memory
- Resolution is O(1) map lookup via a custom `aria-ref` selector engine
- If ref is stale, returns helpful error: "Element with ref 'e52' is not connected to the DOM. Re-take the snapshot."

We won't replicate this full approach (it requires injected scripts and custom selector engines). Our CDP `DOM.resolveNode` approach works correctly. The improvement here is **centralization and better error messages**.

## Implementation Plan

### Step 1: Create `automation/ref.ts` — centralized ref resolution

Extract a reusable utility:

```typescript
import type { CdpSession } from "./CdpSession";

/**
 * Parse a ref string (e.g. "e123") to a backendNodeId number.
 */
export function parseRef(ref: string): number {
    const backendNodeId = parseInt(ref.replace(/^e/, ""), 10);
    if (isNaN(backendNodeId)) throw new Error(`Invalid ref: "${ref}". Expected format: e123`);
    return backendNodeId;
}

/**
 * Resolve a ref to a CDP remote object ID.
 * Throws with a helpful message if the ref is stale (element removed from DOM).
 */
export async function resolveRef(cdp: CdpSession, ref: string): Promise<string> {
    const backendNodeId = parseRef(ref);
    try {
        const { object } = await cdp.send("DOM.resolveNode", { backendNodeId });
        if (!object?.objectId) {
            throw new Error(`Could not resolve ref "${ref}". The element may have been removed. Re-take the snapshot.`);
        }
        return object.objectId;
    } catch (err: any) {
        if (err.message?.includes("No node with given id")) {
            throw new Error(`Ref "${ref}" is stale — the element is no longer in the DOM. Re-take the snapshot.`);
        }
        throw err;
    }
}

/**
 * Resolve a ref and call a function on the resolved element.
 * The function receives `this` bound to the DOM element.
 */
export async function callOnRef(
    cdp: CdpSession,
    ref: string,
    fn: string,
    returnByValue = false,
): Promise<any> {
    const objectId = await resolveRef(cdp, ref);
    const result = await cdp.send("Runtime.callFunctionOn", {
        objectId,
        functionDeclaration: fn,
        returnByValue,
        awaitPromise: true,
    });
    if (result.exceptionDetails) {
        throw new Error(result.exceptionDetails.exception?.description || "callOnRef failed");
    }
    return returnByValue ? result.result?.value : result;
}
```

## Concerns / Open Questions

1. **Injected-script vs CDP approach (decided):** Playwright uses `Map<ref, Element>` in-page memory for O(1) ref resolution. Their main reason is cross-browser support (CDP is Chromium-only). Since we're Electron-only (always Chromium), CDP `DOM.resolveNode` is fine. The only real risk is cross-frame ID collision when implementing US-374 (iframes) — solved with frame-scoped ref prefixes (e.g., `f0-e123`). No need to switch approaches.

2. **Ref format**: Currently `e123` where 123 is `backendDOMNodeId`. For US-374 (iframes), will need frame-scoped prefixes: `f0-e123` (main frame), `f1-e456` (first iframe). The `parseRef()` function should be designed to support this future extension.

3. **`callOnRef` return value**: The current `callOnRef` in `commands.ts` returns void. The new version should support returning values (for `focusElementByRef` which returns ElementKind). Add `returnByValue` parameter.

4. **`DOM.enable()` not required**: We never call `DOM.enable()` yet `DOM.resolveNode` works. If a future Chromium update enforces it, add one line to CdpSession initialization.
